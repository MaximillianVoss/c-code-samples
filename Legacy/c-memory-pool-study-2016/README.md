# C Memory Pool Study (2016)

Три старых проекта объединены как последовательные попытки реализовать
распределитель памяти поверх заранее выделенного буфера:

- `MemoryManager` - незавершенный C-эскиз свободного списка от 9 ноября 2016;
- `MemoryManagerC++` - эксперимент того же дня на C++;
- `MemoryManagerRC` - более поздняя C-версия с публичным заголовком от
  19 декабря 2016.

Каталог `legacy` хранит 19 исходных и проектных файлов побайтно, с прежними
именами каталогов. Полный реестр из 73 исторических файлов, включая SHA-256,
размеры и исходные NTFS-даты, находится в
`preservation/file-metadata.csv`. IDE-базы, промежуточные файлы и старые
сборки не публикуются повторно; их метаданные сохранены здесь и в приватном
репозитории `MaximillianVoss/legacy-native-projects-archive`.

Исторические реализации намеренно не входят в сборку. Они содержат
неопределенные функции, неверную арифметику указателей, отсутствующие
`return`, выходы за границы буфера и ошибки слияния блоков. Подробности и
хронология приведены в `docs/stage-catalog.md`.

## Современная реализация

`src/memory_pool.c` предоставляет небольшое переносимое C17-ядро:

- работает только внутри переданного пользователем буфера;
- выравнивает служебные данные и возвращаемые адреса для фундаментальных
  скалярных C-типов, включая `long double` и указатели;
- использует first-fit, разделение свободных блоков и двустороннее слияние;
- проверяет переполнение, структуру списка, чужие указатели и double free;
- не использует глобальное состояние и допускает несколько независимых пулов;
- сообщает статистику без скрытых выделений памяти.

Это учебный однопоточный allocator, а не замена системного `malloc`.

## Проверка

```powershell
pwsh -File .\scripts\test.ps1
```

Тесты проверяют выравнивание, сохранность данных, first-fit, разделение и
слияние блоков, исчерпание пула, повторное освобождение, чужие указатели и
сброс состояния.

История исходников публикуется отдельно в ветке
`archive/c-memory-pool-study-2016`; аннотированный тег
`original-c-memory-pool-study-2016` указывает на последний source-only этап.
